Introduction

The automation of the Regression Testing is expected and we IRM had a work shop about Selenium in June 2014. I already created automatic scripts for WebSLT and Protrack with Selenium IDE in the first half of 2013, but they weren't enough to verify their behaviors due to a fewer functions. On the other hand, Selenium WebDriver is more flexible and convenient and it can be used at the Unit Testing framework. I therefore started developing the automatic testing with Selenium WebDriver for WebSLT at Feb 2015.

Objective

To implement the WebSLT automatic testing as the Regression Testing. It must ensure that WebSLT works as expected.

Test Approach

I didn't follow the exact plan and analysis stage because we already have many test cases and I were executing the Regression Testing manually. Therefore, the important thing was to replace them to the automatic ones. However, I'm afraid it may not satisfy all situations we encounter during using WebSLT. For the completeness, it would be better to review manuals and all other documents of WebSLT and add the unimplemented behaviors into test cases if need.

Note that the words from Glossary of ISTQB (International Software Testing Qualification Board) are noted down in some places. You can see the meanings here.

Use Case Testing

About the test of the GUI's behavior, to imitate the anticipated user's behaviors are needed. Thus, the clear procedures are required for the actual testing so I adopted Use Case Testing as the Test Design Technique. All use cases involve test procedures and the expected results. Especially here, one use case is regarded as one test case for the better maintainability, and use cases are categorized into four suites.

Test Suites

As noted above, all use cases (that is to say, test cases) are classified into four test suites. Suite1 manages actions on the base pane of WebSLT, Suite2 is the search criteria popup, Suite3 is the report popup, and Suite4 handles generating reports from the base pane. See below for example.

  • Test Suite1 It verifies the behavior on the base pane including logging in and out. This suite contains tests of handling the toolbar's buttons and the entry list. Here are examples.
    • Logging in with a valid user and then logging out
    • Logging in with an invalid user
    • Showing the status page
    • Opening the search criteria popup and then close it
    • Switching to ALMA portal web and switch back
    • etc.

  • Test Suite2 It opens the search popup and changes values of the search criteria, then executes the search and verifies the results. For example:
    • Searching in the various time Intervals
    • Searching with the various entry types
    • Searching by texts
    • Searching by selecting menus
    • Searching by keywords
    • etc.

  • Test Suite3 It opens the report popup and changes values of the report details, then generates reports. The report with the pdf/html format is opened on the new window. The other formatted reports like csv/twiki are saved at /tmp directory automatically. The way of automatic file loading is defined in the Firefox's profile, which is set at creating a WebDriver instance.
    • Generating a Project report
    • Generating a Weather report
    • Generating an Executions report

  • Test Suite4 It generates reports from the entry list on the base pane. At the testing, it searches by the special search criteria.
    • Generating a generic report
    • Generating a Shift report
    • Generating a Downtime Statistics report
    • Generating an End of Night report
    • Generating a Daily report

Note that (2015-05-21): because of use of the profile, Test Suite3 and Suite4 need to run with Firefox. I've not yet looked into the other browsers whether or not they can handle their profile like Firefox at the moment.

Implementation

At implementation, a test case is described as a method in a class. As the result, some of classes are corresponding to the test suites. In order to identify a web element, a classname which is generated by the ZK framework is utilized. You can see all of them below:
  • Class names on WebSLT (pdf)f 2015-09-15

Class Structure

Classes have a hierarchy. Their relations are described in the following diagram.
  • Class Diagram (pdf) 2015-09-15

Here are the explanation of the each class.

  • Definitions.java
    • Consists of class fields and class methods
    • They are used by ShiftlogToolAutotestSuite[1234] classes
    • Variables according to a local test environment can be defined. For example, the server machine name, a user name, its password and the local customized search criteria are included
  • Order.java and OrderedRunner.java
    • Describe a new annotation, "Order", which controls the execution order
    • A test case must have @Order with a number of ordering
  • Initialize.java
    • Describe the common initialization of ShiftlogToolAutoTestSuite[1234] classes
    • Set values of Implicit timeout of WebDriver, a directory name for saving files and the optional profiles
  • ShiftlogToolAutoTestSuite[1234].java
    • The body of the automatic testing
    • Each of classes is just corresponding to each of test suites
    • One test case is exactly corresponding to one use case and implemented as one method
    • A method as a test case needs @Test and @Order
    • A number which is set at @Order is allocated like the following: Sute1 is 001 to 100, Sute2 is 101 to 200, Suite3 is 201 to 300, and Suite4 is 301 to 400
  • BaseWindowProcedure.java and PopupProcedure.java
    • Called by methods (that is, test cases) in ShiftlogToolAutoTestSuite[1234] classes
    • Describe a detail step in a procedure
    • Methods of WebDriver are not handles directly in this class
  • CommonTools.java
    • Methods are designed for the common use, this means not only WebSLT but also the other web application in ALMA
    • Called by methods in BaseWindowProcedure or PopupProcedure classes
    • WebDriver and the other low actions are handles here
  • Tools.java
    • Description of methods in CommonTools class

Features

User Data

  • Data according to a test environment can be set
    • See also the description of Definitions.java
    • Changeable data: a vald user name and its password, an invalid user name and its password, the server name WebSLT runs, values on search criteria, and so on
    • "Begin" and "End" timestamps in "Interval" can be set as well. These values are important because they affect the search result directly. For example, in case much information is shown on a report, getting a screenshot of it fails due to WebDriverException. I'm afraid WebDriver has a restriction about that. So please be careful to set them.

Usage of JUnit framework

  • @BeforeClass is used as pre-processing
    • Getting instances, setting profile of the browser, creating a log file and logging in WebSLT
  • @AfterClass is used as post-processing
    • Mainly logging out WebSLT
  • Test cases are run in order
    • JUnit run methods with @Test in parallel and it doesn't suit to the Use Case Testing. Therefore, @Order is customized to control the running order
  • assertThat() method is used for the verification if an expected behavior is shown

Handling of WebDriver

  • A Classname is used as a locator for finding a web element
    • A locator for finding a web element utilize a classname which is generated by the ZK framework
    • Every updating of the ZK framework, all classnames handle web elements must be reconfirmed
  • A web element tries to get repeatedly in case an exception happens
    • StaleElementReferenceException, which also notes "Element is no longer attached to the DOM", often comes out at trying to get a web element. This is harmless and happens only at once, that is to say, the next trying succeeds without problem. The countermeasure is enough just trying to get a web element again and it's done in CommonTools.java.
  • Implicit wait is implemented
    • The local test environment, ACA4 STE, has not much data in the database, so that 5 seconds are enough for the implicit wait Definitely, it needs to tune at running on the production database because it has numerous data and would take time to get the search on the database.
  • Explicit wait is needed
    • WebDriverWait() method is needed for ensuring the expected behavior. After a new page was loaded, for example, to get a text from a window often fails It is because it is done before the loading surely finished. In other words, we need to wait until the loading finished completely. Implemented waits confirm a text is loaded/unloaded, a button became enable, a web element became available, and so on.

The Others

  • Logging file is created by each Suite
    • The file names are ShiftlogToolAutotestSuite[1234].log
    • In case an exception happens, the related lines are written in the log file
  • A generated report is saved automatically
    • In case of generating a report, a file dialog is usually appeared to ask if it should be opened or saved. WebDriver, however, cannot handle such type of dialog. Therefore, the profile of FIrefox is changed to save the generated file automatically.

Unhandled issued with WebDriver

  • To manage a tab is not handled
  • To maniputale a file dialog (replying for the asking the file should be opened or saved) is not handled
  • To resize an area doesn't seem to be handled, because Firebug cannot find a web element of an movable bar
